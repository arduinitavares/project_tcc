# Investigation Report: Story Generation Scope Drift

## 1. Where the Root Cause Likely Lives

The scope drift originates from a combination of three layers, not a single file:

| Layer | File(s) | Problem |
|-------|---------|---------|
| **Context Injection** | `deterministic_tool_adapters.py` → `user_story_writer_tool()` | Passes the **entire** `pending_spec_content` (full technical specification) and `compiled_authority_cached` (full authority JSON) to every single story generation call. There is no filtering to scope these documents to the specific `parent_requirement` being decomposed. |
| **Prompt Design** | `user_story_writer_tool/instructions.txt` | The instruction says *"Decompose the parent requirement into vertical slices"* but provides **no mechanism** to tell the LLM what is in-scope vs. out-of-scope for this specific requirement. The LLM receives the full spec and is expected to self-limit — which it consistently fails to do. |
| **Missing Validation** | `user_story_writer_tool/schemes.py` + `tools.py` | The `UserStoryWriterOutput` schema has **no max_length constraint** on `user_stories`, no artifact-stream field, and no scope-match validation. The `save_stories_tool` persists whatever the LLM generates without checking for overlap with existing stories. |

### The Core Mechanism of Failure

When the adapter calls the story writer agent, it builds this payload:

```python
args = {
    "parent_requirement": parent_requirement,             # e.g., "Characterization Tests"
    "requirement_context": resolved_requirement_context,  # Theme + Focus Area + Reasoning (short)
    "technical_spec": technical_spec,                     # ENTIRE spec document (~2000+ tokens)
    "compiled_authority": compiled_authority,              # ENTIRE authority JSON (all scope_themes, all invariants)
}
```

The `compiled_authority` contains **all** `scope_themes` (e.g., "people counting", "operational compliance", "scheduling integration", "notifications", etc.) and **all** invariants for the entire product. The `technical_spec` describes every subsystem. The LLM has no way to know that only a subset of this content is relevant to "Characterization Tests" — so it attempts to cover everything, creating stories for UML diagrams, screenshots, packaging, and other artifact streams.

---

## 2. Why the Current Architecture Allows Scope Expansion

### 2.1 No Scope Boundary in the Input Schema

`UserStoryWriterInput` has four fields: `parent_requirement`, `requirement_context`, `technical_spec`, and `compiled_authority`. There is **no field** for:
- `in_scope_artifacts` (what this story group should cover)
- `out_of_scope_artifacts` (what belongs to other story groups)
- `artifact_stream` (test, documentation, analysis, packaging, etc.)
- `existing_stories` (what has already been generated for this milestone)

### 2.2 No Artifact Ownership Model

The `RoadmapRelease` schema defines milestones with `items` (a `List[str]` of requirement names). Each `item` is a free-text string like "Characterization Tests" or "UML Diagrams". There is **no structured decomposition** of a milestone into artifact streams. When a requirement name like "Characterization Tests" is passed to the story writer, the LLM has no metadata telling it that UML diagrams, screenshots, or packaging are separate requirements owned by different story groups.

### 2.3 Full-Document Context Injection

In `deterministic_tool_adapters.py` (line 490-491):
```python
technical_spec = _as_text(state.get("pending_spec_content")).strip()
compiled_authority = _as_text(state.get("compiled_authority_cached")).strip()
```

These are global product-level documents. Every story generation call gets the same full documents regardless of which individual requirement is being decomposed. The `_derive_requirement_context()` function extracts only `theme`, `focus_area`, and `reasoning` from the roadmap release — but this is a 1-2 sentence summary, far too weak to constrain the LLM against the much larger technical_spec and compiled_authority.

### 2.4 No Post-Generation Deduplication

The `save_stories_tool` in `tools.py` does upsert by `(product_id, source_requirement, refinement_slot)`, which prevents duplicate rows **within the same requirement**. But it does **not** check whether the generated story content overlaps with stories already saved under **other requirements**. For example, if "UML Diagrams" were already generated as stories under requirement A, the system will happily generate UML-related stories again under requirement B.

### 2.5 No Story Count Guard

The `UserStoryWriterOutput` schema requires `min_length=1` on `user_stories` but has **no upper bound**. The instructions say *"Do NOT target a fixed number of stories. Continue decomposing until every story is Small."* This is an open invitation for the LLM to produce 8-10+ stories, especially when it has the entire spec as context.

### 2.6 INVEST Score Inflation

The `invest_score` is a self-reported `Literal["High", "Medium", "Low"]` with no external validation. The LLM assigns its own scores. The instruction defines "Small = ~2-3 days" but provides **no calibration heuristics** for documentation-type tasks. A task like "Create a UML class diagram" is inherently small, but the LLM, seeing the full spec complexity, marks it as High effort because it considers the entire system scope.

## 3. Concrete Changes to Fix Scope Drift

### 3.1 Add Scope Boundary Fields to the Input Schema

**File:** `orchestrator_agent/agent_tools/user_story_writer_tool/schemes.py`

Add explicit scope fields to `UserStoryWriterInput`:

```python
class UserStoryWriterInput(BaseModel):
    parent_requirement: Annotated[str, Field(...)]
    requirement_context: Annotated[str, Field(...)]
    technical_spec: Annotated[str, Field(...)]
    compiled_authority: Annotated[str, Field(...)]

    # NEW: Scope enforcement fields
    in_scope_artifacts: Annotated[
        List[str],
        Field(
            default_factory=list,
            description=(
                "Explicit list of artifact types this story group MUST cover. "
                "E.g., ['characterization tests']. Stories outside this list are forbidden."
            ),
        ),
    ]
    out_of_scope_artifacts: Annotated[
        List[str],
        Field(
            default_factory=list,
            description=(
                "Artifact types that belong to OTHER story groups and MUST NOT "
                "appear in this decomposition. E.g., ['UML diagrams', 'screenshots', 'packaging']."
            ),
        ),
    ]
    existing_stories_summary: Annotated[
        str,
        Field(
            default="",
            description=(
                "Summary of stories already generated for this milestone under "
                "other requirements. Used for deduplication."

        ),
    ]
```

## SCOPE ENFORCEMENT (MANDATORY)

### Artifact Boundary Rules
* If `in_scope_artifacts` is provided and non-empty, you MUST generate stories

### 3.5 Add Post-Generation Validation

**File:** New file `orchestrator_agent/agent_tools/user_story_writer_tool/validators.py`

```python
def validate_story_scope(
    stories: List[UserStoryItem],
    in_scope_artifacts: List[str],
    out_of_scope_artifacts: List[str],
) -> List[str]:
    """Return list of violation descriptions for stories that breach scope."""
    violations = []
    out_scope_lower = [a.lower() for a in out_of_scope_artifacts]

    for story in stories:
        combined_text = f"{story.story_title} {story.statement}".lower()
        for artifact in out_scope_lower:
            if artifact in combined_text:
                violations.append(
                    f"Story '{story.story_title}' references out-of-scope "
                    f"artifact '{artifact}'"
                )
    return violations
```

Call this in `story_runtime.py` after parsing the output and before returning success. If violations are found, either filter those stories out automatically or return `is_complete: false` with the violations as feedback.

### 3.6 Improve INVEST Effort Calibration

**File:** `orchestrator_agent/agent_tools/user_story_writer_tool/instructions.txt`

Add calibration heuristics:

```
### INVEST Score Calibration for Effort (Small = ~2-3 days)
* **Documentation tasks** (writing reports, creating diagrams, capturing screenshots):
  - INVEST score should almost always be "High" (easy to do, clearly scoped).
  - Only mark "Medium" if the documentation requires complex analysis first.
  - Never mark "Low" for documentation unless scope is genuinely ambiguous.
* **Test creation tasks** (writing tests, setting up test fixtures):
  - If testing a well-defined behavior: "High"
  - If testing requires significant exploration or setup: "Medium"
* **Integration tasks** (connecting two systems):
  - Typically "Medium" due to interface complexity.
  - "Low" only if the integration spec is vague or missing.
* Remember: invest_score reflects INVEST compliance quality, not effort magnitude.
  A simple well-defined task should be "High" (good INVEST compliance), not "Low".
```

## Summary

The root cause is a **context flooding problem**: the story writer receives the entire product specification and authority document with no boundaries, and the prompt lacks explicit scope enforcement rules. The LLM — following its instruction to "decompose until every story is Small" — interprets the full context as scope and produces stories for artifacts that belong to sibling requirements.

The fix is a three-layer defense:
1. **Input-level**: Tell the LLM explicitly what is in-scope and out-of-scope (schema + prompt).
2. **Injection-level**: Filter the context documents to only include material relevant to the current requirement (adapter logic).
3. **Output-level**: Validate generated stories against scope boundaries before persisting (post-generation validator).
